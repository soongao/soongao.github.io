# Generated RPC and Protocol Models — abase

## 模块定位

`kitex_gen/abase` 是由 `idl/abase/abase.thrift` 生成的 Kitex/Thrift 协议模型包，定义 Abase 服务的枚举、请求/响应结构体、服务接口和 RPC 参数/结果包装类型。核心文件是 `kitex_gen/abase/abase.go`，配套的 `kitex_gen/abase/k-abase.go` 提供 Kitex 快速编解码、长度计算、深拷贝和 `Base`/`BaseResp` 懒初始化辅助逻辑。

这些文件是生成代码，文件头明确标记 `Code generated by thriftgo ... DO NOT EDIT`。需要变更字段、方法或枚举时，应修改 `idl/abase/abase.thrift` 并重新生成。

## 服务接口

`AbaseService` 是本包暴露的 RPC 契约：

```go
type AbaseService interface {
	Get(ctx context.Context, req *GetRequest) (*GetResponse, error)
	Set(ctx context.Context, req *SetRequest) (*SetResponse, error)
	Del(ctx context.Context, req *DeleteRequest) (*DeleteResponse, error)

	BatchGet(ctx context.Context, req *BatchGetRequest) (*BatchGetResponse, error)
	BatchSet(ctx context.Context, req *BatchSetRequest) (*BatchSetResponse, error)
	BatchDel(ctx context.Context, req *BatchDeleteRequest) (*BatchDeleteResponse, error)

	IncrBy(ctx context.Context, req *IncrByRequest) (*IncrByResponse, error)
	BatchIncrBy(ctx context.Context, req *BatchIncrByRequest) (*BatchIncrByResponse, error)

	Append(ctx context.Context, req *AppendRequest) (*AppendResponse, error)
	BatchAppend(ctx context.Context, req *BatchAppendRequest) (*BatchAppendResponse, error)

	XGet(ctx context.Context, req *XGetRequest) (*XGetResponse, error)
	XSet(ctx context.Context, req *XSetRequest) (*XSetResponse, error)
	BatchXGet(ctx context.Context, req *BatchXGetRequest) (*BatchXGetResponse, error)
	BatchXSet(ctx context.Context, req *BatchXSetRequest) (*BatchXSetResponse, error)
```

## 协议模型分组

### 基础 KV 操作

`GetRequest` / `GetResponse` 用于读取单个键：

- `GetRequest.Key`、`GetRequest.TableName` 是必填二进制字段。
- `TimeoutMillisec`、`Priority`、`Flags`、`RetryInfo`、`Base` 是可选字段。
- `GetResponse.ErrCode` 表示请求结果，`Value` 是返回值，`TtlInMillisec` 是剩余 TTL，`RetryInfo` 用于重试路由信息。

`SetRequest` / `SetResponse` 用于写入单个键：

- `SetRequest.Key`、`TableName`、`Value` 必填。
- `TtlType` 与 `Ttl` 同时设置时才表达过期时间。
- `ExistType` 当前只有 `ExistType_ExistNX`。
- `SetResponse.ErrCode` 是写入结果。

`DeleteRequest` / `DeleteResponse` 用于删除单个键：

- `DeleteRequest.Key`、`TableName` 必填。
- `DeleteResponse` 同时包含 `ErrCode` 和 `RespCode`。

### 批量操作

批量模型都采用相同结构：请求侧用 `Reqs` 包装单条请求列表，响应侧用 `Rsps` 包装单条响应列表，并通过 `RespCode BatchResponseCode` 表示批量整体状态。

包括：

- `BatchGetRequest` / `BatchGetResponse`
- `BatchSetRequest` / `BatchSetResponse`
- `BatchDeleteRequest` / `BatchDeleteResponse`
- `BatchIncrByRequest` / `BatchIncrByResponse`
- `BatchAppendRequest` / `BatchAppendResponse`
- `BatchXGetRequest` / `BatchXGetResponse`
- `BatchXSetRequest` / `BatchXSetResponse`
- `BatchExpireRequest` / `BatchExpireResponse`

`BatchResponseCode_Error` 不等价于每条请求都失败；调用方仍需要逐项检查 `Rsps` 中每个响应的 `ErrCode`。`NotFound` 在 IDL 注释中不被视为批量错误。

### 数值与追加操作

`IncrByRequest` / `IncrByResponse` 用于原子增量：

- `Key`、`TableName`、`Delta` 必填。
- 可选 `TtlType`、`Ttl`、`TimeoutMillisec`、`Base`。
- `IncrByResponse.Value` 返回操作后的数值。

`AppendRequest` / `AppendResponse` 用于追加或前插二进制值：

- `Key`、`TableName`、`Value` 必填。
- `Prepend` 为可选布尔字段，设置后表达前插语义。
- `AppendResponse.Len` 返回写入后的长度。

### 带 generation 的 X 系列

`XGetRequest` / `XGetResponse` 和 `XSetRequest` / `XSetResponse` 用于带 `Generation` 的读写路径：

- `XGetResponse.Generation` 返回当前版本。
- `XSetRequest.Generation` 是必填字段，用于服务端校验。
- `XSetResponse.Generation` 返回更新后的版本。
- `ErrorCode_InvalidGeneration` 表示 generation 不匹配。

这组模型通常用于需要 CAS 或版本冲突处理的场景。

### 过期与扫描

`ExpireRequest` / `ExpireResponse` 用于修改过期时间：

- `Key`、`TableName`、`TimeUnit` 必填。
- `TtlOrTs` 表示相对 TTL 或绝对时间戳，含义由 `TimeUnit` 决定。
- `TimeoutMillisec` 和 `Base` 可选。

`ScanRowRequest` / `ScanRowResponse` 用于扫描一行或一组键值：

- `ScanRowRequest.Key`、`TableName`、`Limit` 必填。
- `StartPoint` 是默认字段，不是指针；未设置时为零值 `nil`。
- `Offset`、`TimeoutMillisec`、`Priority`、`IoPriority`、`Base` 可选。
- `ScanRowResponse.Resps` 是 `[]*ScannedKV`，每个 `ScannedKV` 包含 `Key`、`Value`、`TtlInMillisec`。

## 枚举与数据库接口

本包生成的枚举包括：

- `ErrorCode`：请求错误码，例如 `ErrorCode_Success`、`ErrorCode_NotFound`、`ErrorCode_InvalidGeneration`、`ErrorCode_ServiceUnavailable`。
- `TimeUnit`：TTL 单位，包括 `Second`、`MilliSecond`、`AbsoluteTimeSecond`、`AbsoluteMilliSecond`。
- `ExistType`：当前只有 `ExistType_ExistNX`。
- `Priority`：读路由优先级，例如 `ReadMasterOnly`、`ReadSlavePreferred`。
- `IoPriority`：IO 优先级，包括 `Normal`、`Low`、`Idle`。
- `Flags`：读取标志，包括 `NoCacheInsert` 和 `HeaderOnly`。
- `BatchResponseCode`：批量整体状态，`Error` 或 `Success`。

每个枚举都生成了固定辅助方法：

```go
func (p ErrorCode) String() string
func ErrorCodeFromString(s string) (ErrorCode, error)
func ErrorCodePtr(v ErrorCode) *ErrorCode
func (p *ErrorCode) Scan(value interface{}) error
func (p *ErrorCode) Value() (driver.Value, error)
```

其他枚举也遵循同样模式。`Scan` 和 `Value` 让这些枚举可以作为 `database/sql` 字段读写；底层存储值是 `int64`。

## 读写协议执行流

每个结构体都实现 `Read(iprot thrift.TProtocol)` 和 `Write(oprot thrift.TProtocol)`。

`Read` 的执行模式固定：

1. 调用 `apache_warning.WarningApache("TypeName")`。
2. `ReadStructBegin` 进入结构体。
3. 循环 `ReadFieldBegin` 读取字段 id 和 thrift 类型。
4. 根据字段 id 分派到 `ReadFieldN`。
5. 类型不匹配或未知字段调用 `iprot.Skip(fieldTypeId)`。
6. 对必填字段检查 `isset...` 标记，缺失时返回 `thrift.INVALID_DATA`。
7. 所有错误通过 `thrift.PrependError` 补充结构体名和字段名。

`Write` 的执行模式也固定：

1. 调用 `apache_warning.WarningApache("TypeName")`。
2. `WriteStructBegin("TypeName")`。
3. 依次调用 `writeFieldN`。
4. 可选字段先通过 `IsSet...()` 判断指针是否非空。
5. 写入 `WriteFieldStop` 和 `WriteStructEnd`。

必填字段在 `Write` 时不会额外检查零值。例如 `GetRequest.writeField1` 会直接写 `p.Key`。调用方构造请求时必须保证必填字段语义正确。

## 可选字段与默认值模式

生成代码用指针表达 optional 字段：

```go
type GetRequest struct {
	TimeoutMillisec *int64
	Priority        *Priority
	Flags           *Flags
}
```

对应访问方法分三类：

- `IsSetTimeoutMillisec()`：判断字段是否真正出现在对象中。
- `GetTimeoutMillisec()`：字段未设置时返回包级默认变量 `GetRequest_TimeoutMillisec_DEFAULT`。
- `SetTimeoutMillisec(val *int64)`：直接设置指针值。

这意味着 `Get...()` 返回零值不代表字段曾经被设置。需要区分“未设置”和“设置为 0”时必须使用 `IsSet...()`。

枚举可选字段可以用生成的指针函数构造：

```go
priority := abase.PriorityPtr(abase.Priority_ReadMasterOnly)

req := &abase.GetRequest{
	Key:       []byte("object-id"),
	TableName: []byte("table"),
	Priority:  priority,
}
```

## Base 与 BaseResp

请求模型通常包含：

```go
Base *base.Base `thrift:"Base,255,optional" ...`
```

响应模型通常包含：

```go
BaseResp *base.BaseResp `thrift:"BaseResp,255,optional" ...`
```

普通编解码路径中，`ReadField255` 会调用 `base.NewBase()` 或 `base.NewBaseResp()`，再调用对应的 `Read`。快速路径中，`k-abase.go` 还生成了 `GetOrSetBase()` 和 `GetOrSetBaseResp()`：

```go
func (p *GetRequest) GetOrSetBase() interface{}
func (p *GetResponse) GetOrSetBaseResp() interface{}
```

这些方法在字段为空时创建 `base.Base` 或 `base.BaseResp`，用于 Kitex 运行时统一注入和读取基础元信息。

## Kitex 快速路径

`k-abase.go` 是同一套模型的性能补充代码。每个主要结构体都生成：

- `FastRead(buf []byte) (int, error)`
- `FastReadFieldN(buf []byte) (int, error)`
- `FastWrite(buf []byte) int`
- `FastWriteNocopy(buf []byte, w thrift.NocopyWriter) int`
- `BLength() int`
- `DeepCopy(s interface{}) error`

普通 `Read` / `Write` 基于 `thrift.TProtocol`，适用于通用协议路径；`FastRead` / `FastWriteNocopy` 基于字节缓冲和 nocopy writer，供 Kitex 高性能编解码路径调用。业务代码通常不直接调用这些方法，但在新增字段或排查序列化问题时，需要同时关注 `abase.go` 和 `k-abase.go` 是否由同一次 IDL 生成保持一致。

`DeepCopy` 会按字段递归复制结构体、切片、嵌套请求响应以及 `base.Base` / `base.BaseResp`。调用图中可以看到它会创建目标结构体，例如 `NewBatchExpireRequest()`、`NewBatchXGetResponse()`，再复制字段值。

## RPC Args/Result 包装类型

每个服务方法都有对应的 `AbaseService...Args` 和 `AbaseService...Result`：

```go
type AbaseServiceGetArgs struct {
	Req *GetRequest `thrift:"req,1" ...`
}

type AbaseServiceGetResult struct {
	Success *GetResponse `thrift:"success,0,optional" ...`
}
```

这些类型不是业务请求本身，而是 Thrift RPC 方法调用层的包装结构。`abaseservice` 客户端和服务端分发代码会使用它们：

- 请求进入 RPC 时包装成 `AbaseServiceGetArgs`。
- 响应从 handler 返回后包装成 `AbaseServiceGetResult`。
- `ReadField1` 通常创建具体请求，例如 `NewGetRequest()`。
- `ReadField0` 通常创建具体响应，例如 `NewGetResponse()`。
- `SetSuccess(x interface{})` 通过类型断言设置 `Success` 字段。

## 与仓库其他部分的关系

`kitex_gen/abase/abaseservice` 是本模块最直接的消费者。它生成：

- `client.go`：面向调用方的 `kAbaseServiceClient`，方法签名使用本包请求/响应模型。
- `server.go`：服务注册入口，`RegisterService` 绑定 `AbaseService` 实现。
- `abaseservice.go`：底层 `kClient`、方法元信息、handler 分发、args/result 编解码。

`kitex_gen/base` 是本模块的公共依赖，提供 `base.Base` 和 `base.BaseResp`。所有字段 id 为 `255` 的 `Base` / `BaseResp` 都通过该包读写。

仓库中的 `fuxi/client/abase` 当前使用的是外部 `code.byted.org/videoarch/object_data_access/kitex_gen/...` 包，不是这个 `code.byted.org/videoarch/compound/kitex_gen/abase` 包。排查调用链时需要先确认导入路径，避免把两个 Abase 生成模型混用。

## 使用与维护注意事项

不要手工编辑 `kitex_gen/abase/abase.go`、`k-abase.go` 或 `abaseservice` 下的生成文件。字段、枚举、方法签名应从 `idl/abase/abase.thrift` 修改后重新生成。

新增 optional 字段时，调用方应使用 `IsSet...()` 判断是否设置，而不是只看 `Get...()` 的返回值。

新增 required 字段会影响反序列化兼容性。生成的 `Read` 会在字段缺失时返回 `thrift.INVALID_DATA`，旧客户端或旧服务端可能无法互通。

处理批量响应时不要只看 `RespCode`。`BatchResponseCode_Error` 只表示至少存在失败项，最终业务判断需要遍历 `Rsps` 并检查每个响应的 `ErrCode`。

需要传递 `Base` 或读取 `BaseResp` 时优先使用已有字段和生成方法，不要在业务层重新定义并行元信息结构。
